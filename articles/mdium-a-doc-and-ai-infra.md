## はじめに

この記事は、[「生成AI時代のドキュメント基盤」](https://zenn.dev/nuits_jp/articles/2026-03-19-genai-documentation-foundation) に触発されて書いています。（95%Claude Codeの補助です）

こちらの記事を読んで、強く膝を打ちました。生成AI時代において本当に効くのは、派手なエージェントの仕掛けよりも前に「**手元の Office / PDF 資産を、どれだけ情報ロスなく AI に渡せるか**」という地味な土台の部分だ、と。

そして同時に、現場で感じているモヤモヤがあります。

> SharePoint になんでもかんでもデータを突っ込めば、Copilot でそれなりに RAG はできて便利。
> ……でも、その先に待っているのは **麻薬のようなベンダーロックイン**だ。

便利なのは間違いありません。でも、社内のナレッジ資産が特定ベンダーのクラウドと検索基盤に深く依存していくと、後戻りが難しくなります。気持ちよく依存していくほど、抜け出すコストは上がっていく。

だから、**OSS で**、**社内の誰もが使える**、**「人間にも AI にもフレンドリー」なデスクトップアプリ**として、自分の手元にそれが欲しかった。なければ作る。それが **MDium** (エムディウム) です。

https://github.com/straygizmo/mdium

## MDium とは

MDium は、**Markdown を中心に据えたドキュメント基盤デスクトップアプリ** (Windowsのみ対応)です。

- **デスクトップアプリ**：[Tauri](https://tauri.app/)（Rust バックエンド）+ React 19 + TypeScript
- クラウドに資産を吸い上げる必要がない。**手元 / 社内ファイルサーバーのまま**扱える
- 「Office/PDF ⇄ Markdown」の変換を軸に、RAG・Agentic RAG・スライド生成・VBA 編集支援などを束ねる

思想はシンプルです。**Markdown を共通言語にする**こと。この部分については冒頭の記事の「**ドキュメント基盤要件**」を大いに参考にさせていただきました。
MDiumでは、ドキュメント基盤に加えて「**AI基盤※**」を統合することが重要だと考えました。

なぜなら、ドキュメント基盤にAI基盤を統合することで以下のような利点があるからです。

- Agentic RAGが容易になる
- ドキュメントの変換や編集が容易になる
- 複数のドキュメントを横断した分析/生成が容易になる

MDiumでは、ベンダーフリーなAI基盤として [opencode-sdk](https://opencode.ai/docs/ja/sdk/) を採用しました。opencode-sdk については後述します。

※ 本記事では、チャットによる指示でなんでもこなしてくれるClaude DesktopやClaude CodeのようなAIエージェントツールを「AI基盤」と呼ぶことにします。

## 1. 生成AI時代の核心は「情報ロスなく AI に渡す」こと

生成AI時代において重要なのは、従来の Office / PDF データを **どれだけ情報ロスなく AI に与えられるか**です。どんなに賢いモデルでも、入力時点で表が崩れ、見出し階層が消え、数式やマクロが落ちていたら、出てくる答えはそのぶん劣化します。

ここで多くの人が最初に手に取るのが [Markitdown](https://github.com/microsoft/markitdown) でしょう。あらゆるフォーマットを Markdown に変換してくれる便利なツールですが、手元のWord/Excelデータでの変換効率 (変換後の可読性や情報ロスのなさ) が期待したほどではありませんでした。

そこで MDium は、「1つの万能変換器」ではなく、**フォーマットごとに最適な OSS を組み合わせて、可能な限りロスなく Markdown 化する**方針を取りました。

| フォーマット | 変換方向 | 採用 OSS / ライブラリ |
| --- | --- | --- |
| **DOCX → MD** | 読み込み | [mammoth](https://github.com/mwilliamson/mammoth.js) で DOCX を解析 → [turndown](https://github.com/mixmark-io/turndown) + GFM プラグインで HTML→MD。画像も `_images/` に抽出 |
| **XLSX/XLSM → MD** | 読み込み | **[xlsx2md（igapyon 氏）](https://github.com/igapyon/xlsx2md)** を vendoring。表検出・説明文抽出・画像/図形/リッチテキストまで拾う |
| **PDF → MD** | 読み込み | [pdfjs-dist](https://github.com/mozilla/pdf.js) でテキスト抽出 → Y 座標で行をグルーピングし、フォントサイズ比から見出し階層を推定 |
| **MD → PDF** | 書き出し | Markdown プレビューの描画結果を [html2pdf.js](https://github.com/eKoopmans/html2pdf.js)（html2canvas + jsPDF）で PDF 化。 |
| **MD → DOCX** | 書き出し | Rust 製 `docx` クレートで Word 生成 |

特に **xlsx → md** については、**[xlsx2md](https://github.com/igapyon/xlsx2md) に大いに助けられています**。Excel は帳票・台帳・申請書など「実際の業務知識が詰まっている割に、AI に渡しづらい」筆頭フォーマットです。ここを高い精度で Markdown 化できることが、後段の RAG / Agentic RAG の質をそのまま底上げします。igapyon 氏に感謝。

なお **MD → PDF** は、別ロジックで組版し直すのではなく、**画面で見ている Markdown プレビューの描画結果をそのまま PDF に焼き込む**方式です。テーブル・画像・コードブロック、さらに Mermaid 図や数式まで「プレビューで見えたとおり」に出力されるので、PDF だけ見た目が崩れる、という事故が起きにくいのが利点です。長文では `<!-- pagebreak -->` を差し込めば任意の位置で改ページでき、エディタのツールバーからワンクリックで挿入できます。

> 補足：プロキシ対応のため、依存はすべてアプリ内に内包しております。バンドルサイズよりも「社内の誰もが使いやすい」を優先する設計上の判断です。

## 2.「RAG と Agentic-RAG の違いはわかった。で、どうやるの？」に答える

RAG と Agentic RAG の概念解説は世にあふれています。でも実際に詰まるのは、

> 「違いはわかった。**で、結局どうやって自分の社内フォルダでやるの？**」

という最後の一歩。MDium はここを埋めます。

### サーバー上のフォルダ（UNC パス）に対して、簡単に Agentic RAG

社内ファイルサーバーの共有フォルダ（`\\server\share\...` のような **UNC パス**）を指定するだけで、その配下に対して **Agentic RAG** をかけられます。事前にどこかへアップロードして取り込む、という儀式は要りません。**いまそこにあるフォルダ**を対象にできます。

仕組みとしては、各ワークスペースごとに [opencode](https://github.com/sst/opencode) のエージェントを立て、AI 自身がフォルダを探索・読解しながら答えにたどり着きます（後述の opencode-sdk 内蔵）。「埋め込みインデックスを事前構築 → 検索」という従来 RAG とは別ルートで、**AI がエージェントとしてファイルを巡る**スタイルです。

### 従来の（埋め込みベースの）RAG ももちろんできる

事前にベクトルインデックスを作って検索する、オーソドックスな RAG も用意しています。

- 埋め込みモデルは **ruri-v3 系 / e5 系**から選択可能（既定は `ruri-v3-30m`）
- インデックスは SQLite にモデル別（`rag_{model}.db`）で保存。サブフォルダの DB を束ねることも可能

日本語資産が多い現場なら [ruri-v3](https://huggingface.co/sirasagi62/ruri-v3-30m-ONNX) 系、多言語混在なら [e5](https://huggingface.co/Xenova/multilingual-e5-base) 系、といった使い分けができます。
（各リンクはダウンロード元）

### ベクトル + BM25 の「ハイブリッド Agentic-RAG」も用意した

さらに欲張って、**ベクトル検索と BM25（全文検索）を半分ずつ使うハイブリッド検索**も実装しました。

- BM25 は SQLite FTS5（trigram トークナイザ）
- ベクトルはコサイン類似度
- 両者を **RRF（Reciprocal Rank Fusion）** で融合し、`bm25_weight` で配合比を調整可能

意味的な近さ（ベクトル）とキーワード一致（BM25）の良いとこ取りを狙ったものです。

ただし——正直ベースで書いておきます。**実運用では、ハイブリッドに凝るより、素直に Agentic RAG を回したほうが期待する結果を得やすい場面が多い印象**です。検索アルゴリズムを精緻にチューニングするより、AI にフォルダを歩かせたほうが「人間が探すならこう探す」に近い挙動になりやすい、というのが体感です。MDium はそのどちらも選べるようにしてあるので、対象データで両方試して決めてください。

## 3. opencode-sdk を内蔵

MDium は **[opencode](https://github.com/sst/opencode) の SDK（`@opencode-ai/sdk`）を内蔵**しています。

- ワークスペース（フォルダ）ごとに opencode サーバーをローカルループバックで起動
- AI チャット・Agentic RAG・各種ツール実行の基盤として利用
- プロキシ環境でも詰まらないよう、通信は Tauri の HTTP プラグイン経由でシステムプロキシをバイパス

「AI フレンドリー」の実体は、この opencode 基盤の上に、MDium 独自のドキュメント変換・RAG・VBA などのツールを生やしている、という構造です。

### ハーネス（MCP・スキルなど）を、設定ファイルではなく UI で編集できる

opencode をはじめ、この手のエージェント基盤は強力な反面、**MCP サーバーやスキル、エージェント定義といった「ハーネス」部分を、`opencode.jsonc` や `SKILL.md` などの設定ファイルを手書きで編集して整える**のが普通です。慣れていないと、JSON の書式・ファイルの置き場所・スコープ（グローバル／プロジェクト）でつまずきがちな領域でもあります。

MDium は、ここを**専用の設定 UI から GUI で編集できる**ようにしてあります。ファイルの場所や JSON の構文を意識しなくても、画面上のフォームとチェックボックスでハーネスを組み立てられます。

- **MCP サーバー**：ローカル／リモート両対応で追加・編集・削除・有効/無効の切り替え。組み込みサーバーの選択、JSON からのインポート、接続テスト（利用可能ツール数の表示）まで UI で完結
- **スキル（`SKILL.md`）**：名前・説明・本文（Markdown）をエディタ＋プレビューで編集し、チェックボックスで有効/無効を切り替え。組み込みスキルの追加も可能
- **プラグイン**：npm パッケージ指定で追加・有効/無効の切り替え。変更後は「Restart」ボタンで opencode サーバーを再起動して即反映
- **エージェント定義**：ファイルベース（Markdown）／設定ベースの両方に対応し、プロンプト・モデル・温度・利用ツールなどを編集
- **カスタムツール**：TypeScript/JavaScript でツール本体を記述・編集
- **ルール（`AGENTS.md`）・コマンド**：Markdown エディタ＋プレビューで直接編集

いずれも**グローバル設定とプロジェクト（ワークスペース）設定のスコープを切り替え**ながら編集でき、両者の間で移動・コピーもできます。設定ファイルをベースとしたハーネスのチューニングは開発者であっても手を焼く部分かと思います。そのため「社内の誰もが使える」とまではいかないまでも、これらを UI で触れるようにしておくことは重要だと考えています。

## 4. xlsm のマクロを `.bas` に展開し、AI に VBA を書かせる

JTCの現場ではいまだにExcel/VBAが現役で活躍しているのではないでしょうか。
何十年も引き継いだ秘伝のVBAソースをAIに読み書きさせてそれをコピペする作業を繰り返しているとしたら、MDiumはあなたのためのツールです。

`.xlsm`（マクロ入り Excel）の中の **VBA マクロを `.bas` テキストファイルとして展開**し、**AI が直接 VBA を編集**できるようにしました。編集が終わったら、**ボタン一つで `.xlsm` に書き戻し**ます。

- VBA の抽出/注入は MS-OVBA 仕様（2.4.1 の圧縮アルゴリズム）に準拠して実装
- 標準モジュール / クラスモジュール / ドキュメントモジュールに対応
- 書き戻し時は元ファイルをバックアップ

そして重要なのが、**AI は VBA がそれほど得意ではない**という現実への対処です。放っておくと、他言語の構文を混ぜたり、予約語を変数名に使ったり、行継続文字を忘れたりします。

そこで MDium は **VBA 編集スキルをツールに内蔵**しました。AI が VBA を書くときに、よくある失敗（予約語との衝突、行継続、演算子の取り違え、他言語由来の構文混入など）を踏まないようガイドします。「AI に VBA を書かせる」を、実用に耐えるところまで持っていくための仕掛けです。

ただし、モジュール増減時の取り込みは未対応です。
新規モジュール作成時やリファクタリングに伴うモジュール増減時は、手動でxlsmにモジュールを入れ込む必要がありますのでご注意ください。

## 5. Markdown からスライドを — Slidev 内蔵

MDium は **[Slidev](https://sli.dev/) を内蔵**しており、Markdown からプレゼンスライドを作れます。

- 通常の Markdown を **Slidev 形式に変換するコマンド**を内蔵
- リアルタイムプレビュー（iframe）、テーマ選択、フルスクリーン
- **PDF / PPTX へのエクスポート**に対応

スライド作成についてはまだまだ開発中です。

## 6. ビデオ作成機能（開発中・主要パイプラインは稼働）

Markdown から動画を生成する機能も載せています。現時点で以下について動作します。

- Markdown → シーン分割
- ナレーション原稿の自動生成（LLM 経由）
- **TTS（VOICEVOX）による音声生成**
- 字幕（SRT）生成・キャプション合成
- シーン演出（グラデーション / パーティクル / Three.js / Lottie）
- **MP4 / WebM へのエクスポート**（Playwright レンダリング + FFmpeg）

動画エンジンには Remotion ベースの `open-motion` を vendoring しています。

## おわりに

MDium が大事にしているのは、たった2つです。

1. **情報ロスを最小化して、Office/PDF 資産を AI に渡せること**
2. **OSS のデスクトップアプリとして、「人間 & AI フレンドリー」を実現すること**

SharePoint + Copilot の心地よさは本物です。でも、その心地よさに身を委ねた先のロックインを思うと、**「人間にも AI にもフレンドリーで、かつ自由」な選択肢**が、OSS の世界に一つあってもいい。MDium は、その一つでありたいと願っています。
